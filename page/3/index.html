制作头像、血条框UI

2022-05-02

这里使用UGUI，所有的GUI都是在画布上进行操作的

下面进行操作：

  1. Hierarchy，右键，UI>Canvas，创建Canvas后，会自动生成一个Event System游戏对象，它里面挂接的是事件系统，做一些用户交互的事件
  2. Canvas自带了4个组件。

  • Rect Transform（矩形变换）

  • Canvas：配置整体组件显示模式。

    Screen Space-Overlay：这时默认模式，可以让Unity始终在游戏的上层绘制UI。大多数应用程序使用此模式，因为它们希望UI始终位于最上层以提供所有信息。

    Screen Space-Camera：这种模式在于摄像机对齐的平面上绘制UI。平面的大小确定为始终填充整个屏幕，这样撕扯移动摄像机，平面随摄像机一起移动，从而显示于Overlay图形相同的形状。但是，由于平面是在世界中绘制的，而不是在屏幕上绘制的，因此世界中的对象可以绘制在UI的上层。

    World Space：这种模式可以在世界中的任何位置绘制平面。例如，你可以将此平面作游戏中的计算机屏幕，或者用作墙壁，或者放在角色的上层。这在3D游戏中更有用，因为UI会随着距离变小。

  • Canvas Scaler（缩放方式）
    Constant Pixel Size 固定的像素大小

    Scale With Screen Size 根据显示的大小进行缩放

    Constant Physical Szie 固定物理大小

  • Canvas玩家是否点中画布对象的（比如按钮）

  1. 在Hierarchy中点中Canvas，右键>UI>Image,把它命名为（HealthFrame）这就是一个图片了
  2. 选中想选的精灵，把它从文件夹中拖拽到HealthFrame的Inspector中的Image的第一栏（Source Image）。一个头像框就做好了。（我指的是图片，还有一个大框，那个是相对于游戏屏幕的位置。对了，一开始拖拽精灵进去的时候，精灵大小被改了，按Set Native Szie可以恢复成原来的尺寸。要在场景中通过拖拽改大小，可以按住shift来改，这样比例不会变）
  3. 但是设备不同，显示器不同，分辨率不同，分辨率一改，头像框就会和原先的位置不一样了。
  4. 下面牵扯到锚点的概念，那个白色的风车就是锚点
  5. 拖动白色的框框，有一个是不变的，那就是白色风车的和头像框的相对位置。
  6. 把锚点和头像框设置成同一位置，就可以满足需要。
  7. HealthFrame的Inspector>Rect Transform ,点击框框，可以快速改变锚点的位置

好了，在头像框里，应该有个头像，一层包一层，点击HealthFrame，右键，UI>Image,命名为HeadImage。

  1. 依葫芦画瓢，把头像的精灵拖拽到HeadImage中的Inspector中的Image的第一栏（Source Image），调整大小

但是，发现一个问题，在移动头像框的时候，头像的位置并不会贴合。因为锚点不光控制位置，还会控制图像的大小缩放。

想让头像根据外部容器缩放，需要锚点和头像外框进行匹配。

下面进行操作：

  1. 把锚点的四个风车花瓣包裹头像（网上说得很详细啦，搜Unity锚点就行）

下面开始制作血条

制作血条

首先介绍遮罩的概念

遮罩

两个图像，一个是原图像，一个是遮罩图像。显示的是原图像，遮罩图像控制原图像的显示范围，只显示遮罩区域的原图像。所以可以通过修改遮罩图像的大小，来不失帧地显示原图像的部分区域。

制作遮罩血条的方式是，先添加遮罩，然后把血条作为遮罩层的子元素。

  1. 遮罩血条应该是在头像框架里面，点击HealthFrame，右键，UI>Image。

  2. 出现了一个白色的遮罩层，遮罩无所谓什么颜色，也不需要任何精灵，它只是用来占位的，命名为Mask。调整遮罩层的位置，让它正好把应该在头像框里的位置完全遮住。并且需要把轴心（蓝色的圆圈）移动到最左侧的位置。因为如果把轴心留在中间，若减少百分之二十，那么左右两侧都会减少百分之十，就很怪。

  3. 但是我在移动轴心点的时候遇到一个问题：轴心点无法移动，是灰色的。

    解决办法：在顶部菜单栏找到ToggleToolHandlePosition（切换工具手柄位置），把现在的Center模式点击修改为Pivot模式就可以了

  4. 然后在Mask中建一个子对象，它是真正显示血液血条的子对象。Mask，右键，UI>Image,重命名为HealthBar，然后血条上需要挂精灵，遮罩上不需要挂精灵，因为遮罩本身不显示嘛

  5. 这次显示血条不需要把它恢复成原来大小再缩放，只需要让它填充父对象就可以了。在HealthBar的Inspector的Rect Transform，点击快速调控的面板（那个正方形图标），激活以后，按着alt键不松，点最右下角的方格，这个左右就是填充整个区域。

  6. 还想把锚点位置放在左上角，不用按alt键了，点左上角的正方形。（这样会不随遮罩层的缩放而缩放血条）

  7. 下面添加遮罩的功能。回到Mask，选用Mask组件。选中Mask游戏对象（有白色的遮罩的那个），添加“Mask”组件。不想看到遮罩的图形（一般来说，都不想看到），就把“Show Mask Graphic”取消

  8. 现在，选中Mask，调整它的大小，发现血条也在变化

Unity UI系统

2022-05-02

前言：

目前为止，一直使用Debug.Log在主角受伤时输出当前的生命值。但是玩家不会又Console窗口，因此无法查看最终游戏中的日志。所以，为了给玩家提供反馈，交互式应用测序使用用户界面（简称UI）将图像和文本叠加在场景上来显示信息。这种显示方式也成为抬头显示（HUD）。

最新Unity中的3大UI系统

  • UI工具包

UI toolKit UI工具包是Unity中最新的UI系统。它旨在优化跨平台性能，并基于Web技术。您可以使用UI工具包为Unity编辑器创建扩展，用程序创建时运行UI（如果安装了UI工具包软件包）。

Uinty打算让UI工具包成为新UI开发项目推荐UI系统，但它仍然缺少Unity UI（UGUI）和IMGUI中的一些功能。

一句话总结：

全能UI系统，而且是Unity未来UI发展方向，暂时还缺少部分功能，建议优先使用，遇到不足的，再选择别的UI系统补上。

  • Unity软件包（UGUI）

Unity用户界面（Unity UI）软件包（也成为UGUI）是一个较旧的、基于游戏对象的UI系统，宁可以使用它为游戏和应用程序开发运行时UI。Unity使用组件和Game视图来排列和定位用户界面并设置其样式。它支持高级渲染和文本渲染功能。

现在使用Unity的公司，旧项目中大多数还在使用UGUI，甚至是它的前身nGUI（第三方ui），Ruby’s Adventure教程中，使用的还是UGUI

注意：UGUI不能在Unity编辑器中创建或更改用户界面

UGUI的优势

  1. 所见即所得（WYSIWYG）设计工具
  2. 支持多模式、多摄像机渲染
  3. 面向对象的编程

UGUI基本使用步骤

  1. 创建画布Canves
  2. 在画布Canves中添加UI组件（文本、文本框、按钮等等）
  3. 设置好布局、动画、以及摄像机、特效等
  4. 需要的话，再添加交互事件
  • NGUI

第三方UI系统，需付费（$95）,除非入职公司旧系统，还在使用NGUI，否则不推荐学习，UGUI是模仿NGUI做出来的，已经比NGUI更好用，功能更强。

  • IMGUI
    立即模式图形用户界面（IMGUI）是一个代码驱动的UI工具包，它使用OnGUI函数以及实现它的脚本来绘制和管理用户界面以及实现它的脚本绘制和管理用户界面。宁可以使用IMGUI来创建自定义Inspector，Unity编辑器的扩展以及游戏内调显示。不推荐用于构建运行时UI。
    纯代码UI系统，适合程序员做游戏内部测试系统、控制台的UI

学习建议：Unity的三个都学，首先是UGUI（兼容旧项目），其次是UI toolKit，之后是IMGUI，不建议再学NGUI

如果近期不以Unity就业为目标，建议开始学UI toolKit

旧项目：保持系统一致性为原则，使用UGUI或NGUI

新项目：尽量使用Unity UI ToolKit，这个是Unity主推的方向，用它不能实现的再考虑UGUI

游戏内部测试系统（包括控制台）：Unity UI ToolKit不能实现的话，就使用IMGUI

我的游戏呓语（长期更新）

2022-05-02

我要做的游戏不应该是杂乱无章的，它可能是我灵感的迸发，但是否能对的上他人的电波，也是评判游戏好坏的标准之一。我要做的，不是只有我一个人玩的游戏。

因此，游戏的创作需要有一定的约束，那么即使我创作的游戏无法称为精品，这个约束也会强制它不会成为太差的垃圾。那么，如何获取这个约束呢？它来源于我对信息的总和，各种游戏的总结。

汲取，是应该要做的事。



1. “与剧情紧密切合的症状”

恐惧症与躁狂症

eg.背景：埃及的沙漠石窟，与钻地魔虫相遇

角色相对应的：

  • 攻击欲望、昆虫恐惧症(放大化、极端化)

这些症状应该是剧情发展的动力而不是阻力，不能为了戏剧性的游戏效果而终止游戏。不能因这些症状，而让游戏的发展失控。
可以在角色获得这些症状后，让角色做出与理智时“相似”而又差异鲜明的行为。比如破坏一些本该保护的事物。


2.”塑造更加生动的NPC”

  1. NPC在游戏中存在的意义

    eg.背景：调查员收到老人的委托，希望调查员能将盘踞在他宅邸中可怕的怪物驱逐

老人相对应的作用：

  • 委托的发起者

但也可以是

  • 线索的提供者、幕后真相的隐藏者

但是这仅仅像是一个为了发起任务而发起任务的角色，因此需要对他塑造特点，塑造特点的同时需要考虑游戏的场景是否可以表现角色的特点。

一个角色能为我们带来深刻影响的原因究竟是什么？

  • 性格？
  • 故事？
  • 行为？

能够停留在我们印象中的，往往是一两个支离破碎的场景。塑造一个令人印象深刻的NPC非常关键的地方在于为角色添加一个难以忘记的特点，添加一个难以忘记的方法有很多，可以是一个冷幽默、冷笑话，或者是一个看上去就很离谱的设定或者台词，也可以是虽然普通但是被角色不停使用后逐渐成了代表这个角色的符号。

重复可以带来洗脑的效果，不过最好少用这种方法，容易让人心烦。

巧妙地设计一个外号、动作、场景，在博得玩家会心一笑的同时逐渐建立起这个特征与角色的关系，最终让这样一个特征甚至能代替角色本身。即要为角色创造一个足够被玩家记住的梗。

优秀的剧情反转和设计精巧的悲剧结尾（注意需要是在意料之外，也得在情理之中）。悲剧是表达的手段，悲剧背后的矛盾才是让人难以忘怀的内容

视觉风格-粒子

2022-05-01

粒子系统

粒子系统（Particle System）组件通过在场景中生成和动画化大量小型的2D图像来模拟流动实体。

粒子系统可以创建数十个甚至数百个粒子，这些粒子是很小的图像，具备方向、速度、生命周期、颜色等其他属性。

一个粒子系统可以模拟并渲染许多称为粒子的小的图像或者网格并产生视觉效果。系统中的每个粒子代表效果中单个图形元素。系统共同模拟每个粒子以产生完整效果和印象。

需要创建如火、烟、液体之类动态对象时，由于很难有网格（3D）或精灵（2D）描绘这种对象，所以粒子系统很有用。



选择粒子系统解决方案

  • 内置粒子系统
  • Visual Effect Graph

内置粒子系统比较简单，它解决不了的让Visual Effect Graph来。

内置粒子系统在CPU上模拟粒子行为（Visual Effect Graph是在显卡的GPU上的），这种技术具有以下主要优点：

  1. 可以使用c#脚本与系统及其中的各个粒子进行交互
  2. 粒子系统可以使用Unity的基础物理系统，从而与场景中的碰撞体进行交互

制作粒子系统，首先找到素材，下载的项目中Art>Sprites>Environment>VFX已经有给我们准备的素材了，VFX本身也就是特效的意思。

但那本身是一个完整的图形，首先要对它进行分割。

分割很简单。

素材切割

  1. 先选中素材
  2. Inspector>Sprite Mode选中Multiple
  3. 打开Sprite Editor，用Slice按Grid By Cell Count把它切割成4*4吧。（C、R改成4）

切割好，接下来就是粒子特效的生成

生成粒子特效

  1. Hierarchy右键，Effects>Particle System(按教程上，把它改名为SmokeEffect，添加这个，默认会生成白色的小球)(Particle System Force Field结合Particle System会产生物理效果；Trail是笔记、Line是线)（记得合理利用Particle Effect的那个框）
  2. 添进来以后，会发现SmokeEffect只包含Particle System这个组件，这个组件就是所谓的内置物理系统，下面有非常多的属性可以进行配置。
  3. 要换素材的话，选择（Texture Sheet Animation），打开以后，把Mode由Grid改成Sprites，Sprites就 可以换素材了，默认会有一个Sprite，但是你有几个素材就添加几个，有个“+”号。然后把素材拖进去就可以了。

摄像机跟随和边界

2022-04-30

显示区域是归Unity里的Main Camera管的。目前为止，游戏只在一个屏幕上运行，这时因为用了一个静态摄像机。

摄像机-Cinemachine

无需编写代码，使用Cinemachine的Unity包来自动控制摄像机。摄像机会跟随主角。

对Cinemachine的介绍：Cinemachine是一套用于操作Unity摄像机的模块。Cinemachine解决了跟踪目标、合成、混合和镜头之间剪切的复杂数学和逻辑。它旨在减少开发过程中耗时的手动操作和脚本修订的数量。


我在安装Cinemachine的过程中，遇到了点问题。先是在Pack Manager中安装Cinemachine 2.6.11，可能是版本太高了，在Hierarchy中右键并不会显示Cinemachine，并且按教程中说是顶部菜单栏中也找不到Cinemachine。就安装了2020年，3月10日和我2020.3.30的Unity相近的版本。同样在Hierarchy中右键并不会显示Cinemachine，但是在重启项目后，顶部菜单栏显示了Cinemachine。




Cinemachine>Create 2D Camera,会建成一个摄像机，这个摄像机默认的名字的CM vcam1（CM==Cinmachine，v==虚拟的，cma==Camera）

最终的目标是把摄像机限定在特定的范围内，这样就不显示地图边界外的对象，并且要让摄像机跟随2D目标移动。

Cinemachine可以与多个摄像机配合使用，并根据游戏的需要切换在多个摄像机之间进行切换，例如在对话中对某个镜头与反拍镜头进行切换。可以在每个摄像机中显示不同的设置，告诉实际的摄像机（在Hierarchy中名为Main Camera主摄像机）当前哪个虚拟摄像机处于活动状态，以便复制设置。

修复错误行

上述代码运行后，控制台有两个错误行

  • 1.null 引用异常

双击null引用异常错误，会打卡Projectile脚本并定位到Rigidbody2d.AddForce(direction*force)行，这表示尽管我梦在Start中获得了刚体，但Rigidbody2d变量还是为空（包含null）。

这是因为在创建对象时，Unity不会允许Start，而是在下一帧才开始运行。因此，在飞弹上调用Launch时，只实例化（Instantiate），不调用Start，因此Rigidbody2d仍然为空。要解决此问题，请将Projectile（我的是Projectile1）脚本中的void Start（）函数重命名为void Awake（）。

与Start刚好相反，在创建对象时（调用Instantile时）就会立即调用Awake，因此在调用Launch之间前已经正确初始化Rigidbody2d

tip:在Unity游戏程序运行时，所有附加到游戏对象上的脚本组件都会运行
Awake方法和Start方法都只会运行一次，多用于数据初始化。这两个方法的执行顺序是：先执行Awake方法，再执行Start方法，最后才是Update（）

  • 2.飞弹与Ruby发生了碰撞

因为飞弹是在Ruby上创建的飞弹。一旦创建，物理系统就会告知，飞弹刚体正在与Ruby碰撞体碰撞。然后再OnCollisitionEnter函数中调用Destroy，立刻销毁飞弹。

为了测试一下飞弹是否与Ruby碰撞，不销毁飞弹。但是又会遇到一个问题，那就是碰撞后会立刻停止移动。


图层和碰撞

要修复这个问题，正确方法是使用图层，图层可以讲游戏对象分组在一起，以便可以对它们就行筛选。现在的目标是创建一个角色图层来放入Ruby游戏对象，然后创建一个飞弹图层来放入所以飞弹。

然后告诉物理系统，角色图层和飞弹图层不能碰撞，要物理系统忽略这些图层中的对象之间的所有碰撞。

  • 1.要查看某个游戏对象在哪个图层，单击游戏对象Inspector右上角的Layers下拉选单。所有对象都从Default层开始（即图层编号为0）.游戏最多可以包含32个不同的图层。

  • 2.更改游戏对象的图层后，还需要在项目配置（Project Settings）里的Physics 2D里的Projectile和Character之间的对勾取消选择，这表示它们不会互相碰撞。这些横向和纵向都代表图层。

动画术语

动画剪辑术语

  • 动画剪辑（Animation Clip）：可用于动画角色或简单动画数据。这是一种简单的单位动作，例如“空闲”、“行走”、“奔跑”
  • 动画曲线（Animation Curves）：曲线可以附加到动画剪辑，并由游戏中的各种参数控制
  • Avatar遮罩（Avatar Mask）：为骨架指定要包含或排除哪些身体部位，在动画层或导入器中使用。

Animator和Animator Controller术语

  • Animator组件（Animator Component）：相关动画的东西都是通过这个组件来进行同一的管理，来挂接到游戏对象上
  • Animator Controller：是真正用来管理动画素材的。把Animator Controller放到Animator组件上，整个系统就挂接到一起了。用来控制不同状态下动画的播放，以及状态间的转换。
  • Animator窗口（Animator Window）
  • 动画状态机（Animation State Machine）
  • 动画参数（Animation Blend Tree）

Animator>Parameters->”+”,有些时候Trigger比Bool更好用，Bool进入一个区域后出来要更改Bool值来改变状态，而Tigger不
可以根据时间或参数来决定状态转换。如果勾选了Has Exit Time，Exit Time的值也可以更改了，这时就是根据时间来进行状态转换的条件。这时即使没有在Conditions设置转换条件，到达退出时间，会自动退出当前模式

复刻动画状态的转换，肯定要先看源码。选择已经做好的Ruby的Animator，看关于Idle和Moving之间的转变。
点击Idle，Idle->Moving有个Has Exit Time，取消勾选，会立马转变

关于新词汇：

  • Fixed Duration

解释：如果选中Fixed Duration复选框，则以秒为单位解读过渡时间。如果未选中Fixed Duration复选框，则过渡的时间解读未源状态的标准化时间的一部分。（没懂）

  • Transition Duration（s） 对当前状态持续时间的过渡时间，以标准化或秒为单位（具体取决于Fixed Duration模式），慢慢渐变的一个过程，此时间在过渡图中显示为两个蓝色标记之间的部分，0代表没有重叠。
  • Transition Offset 过渡到目标状态的起始播放的时间偏移。例如，值为0.5表示目标状态在其时间轴的50%出开始播放。
  • Interruption Source 影响它的源
  • BlendTree Parameters> Conditions ，那个框满足它什么条件进行转换。
